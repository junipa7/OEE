3. 네트워크 전송 및 장애 대응 모듈 개발 (Edge ↔ HQ)

1. 모듈 개요 (Module Overview)

본 모듈은 Edge 환경에서 30분 또는 1시간 주기로 메모리에 누적된 통계 데이터(Mean, Variance, KLL Sketch 등)를 직렬화하여 본사 통합 서버로 전송합니다. 글로벌 네트워크의 간헐적 단절에 대비하여 데이터 유실을 100% 방지하는 것이 최우선 목표입니다.

  • 이중 압축: KLL 스케치 자체의 수학적 데이터 압축에 더해, JSON 페이로드 전체를 Zstandard(Zstd) 알고리즘으로 한 번 더 압축하여 트래픽을 극단적으로 최소화합니다.
  • Store & Forward: HTTP 전송 실패 시 로컬 큐(Queue)에 데이터를 안전하게 저장하고, 네트워크 복구 시 순차적으로 재전송(Retry)합니다.

2. 핵심 구현 1: 페이로드 생성 및 Zstd 고속 압축

통계 상태값과 이진 데이터인 KLL 스케치를 안전하게 JSON으로 묶기 위해 Base64 인코딩을 거친 후, Zstd로 최종 압축하는 로직입니다.

import json
import base64
import zstandard as zstd

class PayloadBuilder:
    def __init__(self, corporation_code):
        self.corp_code = corporation_code

    def build_and_compress(self, eqp_id, start_time, end_time, welford_stats, kll_sketch_bytes):
        """통계 데이터를 JSON으로 구성하고 Zstd로 압축"""
        
        # 1. KLL 스케치 바이트 배열을 JSON에 담기 위해 Base64로 인코딩
        encoded_sketch = base64.b64encode(kll_sketch_bytes).decode('utf-8')
        
        # 2. 페이로드 구성
        payload = {
            "corp_code": self.corp_code,
            "eqp_id": eqp_id,
            "period_start": start_time.isoformat(),
            "period_end": end_time.isoformat(),
            "stats": {
                "count": welford_stats['count'],
                "mean": welford_stats['mean'],
                "variance": welford_stats['variance']
            },
            "kll_sketch_base64": encoded_sketch
        }
        
        json_str = json.dumps(payload)
        
        # 3. Zstd 압축 (레벨 3 - 속도와 압축률의 최적 밸런스)
        compressor = zstd.ZstdCompressor(level=3)
        compressed_payload = compressor.compress(json_str.encode('utf-8'))
        
        return compressed_payload

3. 핵심 구현 2: Store & Forward 및 재전송 로직

네트워크 단절 시 데이터를 로컬 DB(SQLite) 큐에 저장하고, 데몬(Daemon) 쓰레드가 주기적으로 재전송을 시도하는 안정성 보장 로직입니다.

import requests
import sqlite3
import time

class NetworkTransmitter:
    def __init__(self, hq_endpoint, db_path="edge_queue.db"):
        self.hq_endpoint = hq_endpoint
        self.conn = sqlite3.connect(db_path, isolation_level=None)
        self._init_queue_table()

    def _init_queue_table(self):
        # 전송 대기열 테이블 생성
        query = """
            CREATE TABLE IF NOT EXISTS TRANSMIT_QUEUE (
                ID INTEGER PRIMARY KEY AUTOINCREMENT,
                PAYLOAD BLOB NOT NULL,
                CREATED_AT DATETIME DEFAULT CURRENT_TIMESTAMP,
                RETRY_COUNT INTEGER DEFAULT 0
            )
        """
        self.conn.execute(query)

    def send_data(self, compressed_payload):
        """데이터 전송 시도. 실패 시 큐에 저장"""
        try:
            headers = {'Content-Type': 'application/zstd'}
            response = requests.post(self.hq_endpoint, data=compressed_payload, headers=headers, timeout=10)
            response.raise_for_status()
            print("전송 성공")
        except requests.exceptions.RequestException as e:
            print(f"네트워크 오류, 큐에 저장: {e}")
            self._save_to_queue(compressed_payload)

    def _save_to_queue(self, payload):
        self.conn.execute("INSERT INTO TRANSMIT_QUEUE (PAYLOAD) VALUES (?)", (payload,))

    def process_queue(self):
        """백그라운드에서 주기적으로 실행되는 재전송 스케줄러"""
        cursor = self.conn.execute("SELECT ID, PAYLOAD FROM TRANSMIT_QUEUE ORDER BY ID ASC LIMIT 50")
        rows = cursor.fetchall()
        
        for row_id, payload in rows:
            try:
                headers = {'Content-Type': 'application/zstd'}
                response = requests.post(self.hq_endpoint, data=payload, headers=headers, timeout=10)
                if response.status_code == 200:
                    # 전송 성공 시 큐에서 삭제
                    self.conn.execute("DELETE FROM TRANSMIT_QUEUE WHERE ID = ?", (row_id,))
            except requests.exceptions.RequestException:
                # 실패 시 재시도 횟수 증가 (알람 처리용)
                self.conn.execute("UPDATE TRANSMIT_QUEUE SET RETRY_COUNT = RETRY_COUNT + 1 WHERE ID = ?", (row_id,))
                break # 통신 장애가 지속 중이므로 루프 중단 후 다음 주기에 시도

4. 모듈 개발 시 체크포인트

  • 보안 (Security): 글로벌 전송 시 데이터 탈취 방지를 위해 반드시 HTTPS (TLS 1.2 이상) 프로토콜을 사용해야 하며, API Key 또는 JWT 토큰을 통한 Edge 인증 로직이 추가되어야 합니다.
  • 큐 모니터링 (Queue Monitoring): 장기간 통신 단절 시 SQLite DB 파일 용량이 비정상적으로 커질 수 있으므로, TRANSMIT_QUEUE의 데이터 건수가 임계치(예: 10,000건)를 초과하면 로컬 알람을 띄우고 오래된 데이터를 Drop할지(혹은 유지할지) 정책적 결정이 필요합니다.
  • HQ 수신부 대응: 본사(HQ) API 서버는 Content-Type: application/zstd 헤더를 인식하고, 역순으로 압축 해제(Zstd Decompression) -> JSON 파싱 -> Base64 디코딩을 수행하는 전용 엔드포인트를 구현해야 합니다.
