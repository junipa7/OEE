📄 설비 생산성 관리 시스템 요구사항 정의서
1. 시스템 개요 및 목적
  - 목적: 7개 해외 법인 설비의 실시간 메시지를 수집·분석하여 **실시간 OEE(설비종합효율)**를 추정하고, 본사에서 통계 처리를
 통해 **Standard Time(표준 시간)**을 산출하는 것을 목적으로 함.

  - 핵심 기술: 대규모 데이터 처리를 위한 KLL 스케치(데이터 압축) 및 Chan 알고리즘(평균/분산 연산) 적용.

2. 인프라 및 데이터베이스 구성
  - 본사 (HQ):
    . Database: Oracle Database 운영.
    . 역할: 글로벌 기준정보 관리 및 전 법인 데이터 통합 통계 분석.
  - 생산법인 (Edge):
    . H/W: 인텔® 코어™ Ultra(시리즈 2), 64GB RAM, 1TB SSD (고성능 Linux 서버).
    . Database: SQLite (가벼운 파일 기반 처리로 I/O 속도 극대화).
    . 역할: 실시간 메시지 파싱, 로컬 OEE 추정 및 일별 실적 집계.

3. 데이터 수집 및 상태 관리 (Context Management)
  설비 메시지의 누락 정보를 보완하기 위해 SQLite 내에 'Current Context' 정보를 유지합니다.
  - 수집 이벤트: Process Start/End, Lot Start/End.
  - MOS 연동 및 보완 정보: * 현재 작업 중인 Lot ID, 생산 기종, 현재 공정.
    . Recipe ID: 작업 조건별 CT 편차 분석용.
    . 투입 및 완성 수량: 실시간 달성률 및 수율 산출용.
  - 설비 상태 정보: 가동/비가동 여부 교차 검증.

4. 주요 기능 요구사항
  A. 생산법인 (Edge) 기능
    - 메시지 해석 및 결합: 설비 로우(Raw) 메시지에 SQLite의 상태 정보를 결합하여 완전한 생산 로그 생성.
    - 실시간 지표 산출: Process 이벤트를 카운트하여 Cycle Time(CT), Tact Time(TT) 및 일별 생산실적 집계.
    - 데이터 압축 연산: 이벤트 발생량이 많은 설비는 KLL 스케치로 압축하여 데이터 부하 최소화.
    - 현장 모니터링: 법인별 실시간 OEE 및 가동 현황 대시보드 제공.

  B. 본사 (Main) 기능
    - 기준정보 배포: Oracle의 표준 코드(기종, 공정, ST)를 JSON/CSV 형태로 법인 SQLite에 주기적 전송.
    - 표준 시간(ST) 산출: 7개 법인에서 수집된 CT, TT 통계 데이터를 바탕으로 최적의 Standard Time 정의.

5. 데이터 전송 및 네트워크 최적화
   해외 7개 법인의 네트워크 환경을 고려하여 전송 부하와 불안정성을 최소화합니다.

    - 30분 단위 배치 전송: 실시간 전송으로 인한 병목을 방지하기 위해 Edge Server에서 집계/압축된 데이터를 30분 주기로 본사에 전송함.
    - 장애 대응 큐(Queue): 네트워크 단절 발생 시 데이터 유실을 막기 위해 SQLite 내 전송 대기 큐를 운영하여 연결 복구 시 재전송 수행.
    - 데이터 최대 압축 전송:
      . 로우 데이터 대신 KLL 스케치 및 Chan 알고리즘으로 요약/계산된 집계치만 전송하여 크기 획기적 감소.
      . 전송 페이로드(JSON/CSV 등)에 Gzip 또는 Zstandard(zstd) 등의 최대 압축 방식을 적용하여 대역폭 사용량 최소화.

📊 데이터 처리 및 전송 흐름 요약

단계              수행 위치     주요 작업                                         전송 전략
1. 기준정보 수신  본사 → 법인   표준 기종/공정 코드를 Oracle에서 SQLite로 동기화    필요 시 (Event-driven)
2. 데이터 보완    법인 (Edge)   설비 메시지 + MOS 상태 정보 결합                   실시간 (Local)
3. 실시간 연산    법인 (Edge)   KLL/Chan 알고리즘 기반 OEE/CT/TT 산출             실시간 (In-memory)
4. 결과 전송      법인 → 본사   집계된 통계 및 실적 데이터 전송                     30분 단위 / 최대 압축


📄 단계별 진행순서
1단계: 상세 설계 (Detailed Design)
      본격적인 개발에 앞서 데이터 구조와 연동 방식을 구체화하는 단계입니다.
  - DB 스키마 및 상태 정보 설계: 본사 Oracle DB와 법인 SQLite의 테이블 스키마를 정의합니다. 특히 설비 메시지의 누락을 보완하기 위한 'Current Context' 테이블의 구조를 상세히 설계합니다.
  - 동기화 및 전송 프로토콜 설계: 본사와 법인 간 데이터 동기화 방식(JSON/CSV 구조 등)을 구체화합니다. 네트워크 부하 최소화를 위한 30분 단위 전송 패키징 규격 및 최대 압축(Gzip/Zstd) 방식을 설계합니다.
  - 데이터 보완(Mapping) 로직 설계: 설비의 Raw 메시지와 MOS 상태 정보(Lot, 기종, Recipe 등)를 결합하는 상세 트리거와 규칙을 설계합니다.
  - 알고리즘 파라미터 최적화 설계: 고성능 서버 자원을 활용하여 KLL 스케치의 압축률과 Chan 알고리즘의 통계 정확도를 결정짓는 세부 파라미터를 설정합니다.

2단계: 시스템 개발 (System Development)
      설계된 규격을 바탕으로 본사와 법인의 각 모듈을 실제 코드로 구현합니다.
  - 본사(HQ) 통합 서버 개발: 글로벌 기준정보를 관리하고 표준 코드(기종, 공정, ST)를 배포하는 기능을 구현합니다. 7개 법인에서 수집된 데이터를 바탕으로 Standard Time(표준 시간)을 산출하는 통계 처리 로직을 개발합니다.
  - 생산법인(Edge) 실시간 처리 모듈 개발: Linux 기반의 고사양 서버 환경에서 설비 메시지를 파싱하고 실시간으로 CT/TT 및 일별 생산실적을 집계하는 모듈을 구현합니다. KLL 스케치 및 Chan 알고리즘을 적용한 데이터 압축 및 실시간 OEE 추정 모듈을 개발합니다.
  - 실시간 상태 관리 모듈 개발: MOS와 연동하여 현재 작업 중인 Lot ID, 투입/완성 수량 등의 정보를 실시간으로 SQLite에 반영하고 메시지와 결합하는 기능을 개발합니다.
  - 네트워크 전송 및 장애 대응 모듈 개발: 30분 주기의 배치 전송 모듈과 네트워크 단절 시 데이터 유실을 방지하는 전송 대기 큐(Queue)를 개발합니다.

3단계: 시각화 및 통합 검증 (Visualization & Validation)
      개발된 모듈들을 연동하고 현장 적용 가능성을 테스트합니다.
  - 대시보드 UI 개발: 7개 해외 법인 현장에서 실시간 OEE 및 가동 현황을 파악할 수 있는 시각화 대시보드를 구축합니다.
  - 통합 정합성 테스트: Oracle과 SQLite 간의 이기종 DB 데이터 동기화가 설계된 프로토콜에 맞게 오차 없이 이루어지는지 검증합니다.
  - 성능 및 부하 테스트: 대규모 설비 이벤트 발생 시 인텔® 코어™ Ultra 및 64GB RAM 환경에서 인메모리 연산(KLL/Chan)이 지연 없이 수행되는지 확인합니다.
