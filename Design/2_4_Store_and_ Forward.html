설비 생산성 관리 시스템
데이터 동기화 및 전송 프로토콜 (Store & Forward) 상세 설계서

본 문서는 전 세계 7개 생산 법인(Edge)에서 산출된 방대한 설비 데이터를 본사(HQ) 통합 서버로 유실 없이, 그리고 네트워크 부하를 최소화하여 전송하기 위한 'Store & Forward' 아키텍처 및 페이로드 규격을 정의합니다.

📡 1. 동기화 아키텍처 (Store & Forward Flow)

현장의 통신 단절에 완벽하게 대비하고 서버 트래픽을 최적화하기 위해, 실시간 스트리밍이 아닌 30분 단위 배치(Batch) 큐잉 방식을 채택합니다.

프로세스 단계 주요 동작 및 설명
1. 수집 및 통계 요약 (Store) Edge 서버에서 30분 동안 생성된 TB_ENRICHED_LOG 데이터를 백그라운드 프로세스가 스캔하여, KLL/Chan 알고리즘 기반의 통계치로 요약 압축합니다.
2. 대기열 적재 (Queueing) 요약된 데이터를 JSON 형태로 구성하여 SQLite의 TB_SEND_QUEUE 테이블에 IS_SENT = 'N' 상태로 인서트합니다.
3. 스케줄러 전송 (Forward) 매 30분 정각(예: 00분, 30분)에 전송 스케줄러가 큐(Queue)를 읽어 본사의 REST API 엔드포인트로 HTTP POST 전송을 시도합니다.
4. 장애 대응 및 복구 (Retry) 네트워크 단절로 전송에 실패할 경우, 데이터는 유실되지 않고 큐에 안전하게 보존됩니다.
통신 복구 시, SEQ_NO가 가장 오래된 미전송 패키지부터 순차적으로 일괄 재전송(Catch-up)하여 본사 DB의 시계열 정합성을 보장합니다.

📦 2. 전송 패키징 규격 (JSON Payload)

요청된 최신 명명 규칙(HEADQUARTERCODE, SITE_CODE, GODS_CODE, PROD_ID)을 반영한 30분 단위 전송 페이로드 규격입니다. 배열(Array) 구조를 사용하여 한 번의 전송에 다수의 설비 데이터를 포함합니다.

🗜️ 3. 3단계 데이터 압축 및 최적화 전략

해외 법인 간의 제한적인 대역폭을 극복하기 위해 데이터 생성부터 네트워크 전송 단계까지 3중 압축 전략을 적용합니다.

Step 1: 데이터 요약 (Algorithm)

원천(Raw) CT/TT 데이터를 모두 전송하지 않고, KLL 스케치(분포도) 및 Chan 알고리즘(평균/분산)으로 30분 단위 통계 특성만 추출하여 부피를 99% 삭감합니다.

➔

Step 2: 직렬화 (Serialization)

생성된 인메모리 KLL 스케치 객체를 Base64 인코딩을 통한 Byte 배열로 변환하여 텍스트 오버헤드 없이 최소한의 크기(수백 바이트)로 JSON에 삽입합니다.

➔

Step 3: 네트워크 압축 (Network)

완성된 JSON 페이로드를 HTTP 전송 직전 Zstandard (Zstd) 알고리즘으로 전체 압축합니다. Zstd는 Gzip 대비 압축/해제 속도가 뛰어나 Edge 환경에 최적화되어 있습니다.

✅ 4. 전송 후 처리 로직 (Post-Transmission)

  • 본사 수신 확인 (Acknowledge): 본사 Oracle 서버가 데이터를 성공적으로 적재한 후 HTTP 200 OK 응답을 반환해야 전송 성공으로 간주합니다.
  • 큐(Queue) 정리: 200 OK 수신 시, 해당 페이로드의 TB_SEND_QUEUE 내 IS_SENT 상태를 'Y'로 변경합니다. (보관 주기 7일 경과 후 하드 삭제 수행)
  • 인메모리 초기화: 전송 성공 확인 즉시, Edge 서버 메모리에 적재되어 있던 해당 30분 구간의 KLL Sketch 객체와 Chan 상태 변수(n, mean, m2)를 초기화(Reset)하여 메모리 누수를 원천 차단합니다.
